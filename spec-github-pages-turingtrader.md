## 1. Context

TuringTrader is a Python-based algorithmic trading system with IBKR integration, a backtesting framework, and a Dash-based monitoring dashboard. The existing dashboard (`monitoring/dashboard.py`) is a server-side Python Dash app that depends on PostgreSQL/TimescaleDB, Redis, and the IBKR trading engine â€” none of which can run on GitHub Pages.

This spec defines a **new client-only SPA** that replaces the Dash dashboard for public-facing use. It will display pre-computed backtest results and algorithm evaluation data, deployed as a static site to GitHub Pages at `https://mazharm.github.io/turingtrader/`.

The existing Dash dashboard, Docker infrastructure, and trading engine remain unchanged â€” they continue to serve as the live monitoring system.

## 2. What the SPA Displays

The SPA shows **backtest and evaluation results** â€” data that is generated by running the Python backtesting scripts and committed as JSON to the repo. It does NOT show live trading data.

### Data sources (already exist in the repo)

The backtesting framework (`backtesting/performance_analyzer.py`, `evaluate_algorithm.py`, `backtest_volatility_*.py`) already produces rich output including:

- Per-risk-level results: total return, annualized return, max drawdown, Sharpe ratio, Sortino ratio, Calmar ratio, win rate, profit factor, best/worst day, best/worst month, profitable months %, trade count
- Daily equity curves per risk level
- Risk level comparison summaries
- Parameter sweep results (IV/HV thresholds, strike widths, target deltas)

### What needs to happen

A new Python script (`scripts/generate_dashboard_data.py`) will run the existing backtesting/evaluation code and serialize the output as JSON files into `dashboard/public/data/`. These JSON files are committed to the repo and consumed by the SPA at runtime â€” no build-time Python execution needed in CI.

## 3. Technology Choices

| Concern | Choice | Rationale |
|---------|--------|-----------|
| Framework | **React 19** with TypeScript | Widely supported, good ecosystem for data dashboards |
| Build tool | **Vite** | Fast builds, simple config, first-class React support |
| Charting | **Plotly.js** (`react-plotly.js`) | Matches the existing Plotly charts in the Python codebase; interactive out of the box |
| Styling | **CSS Modules** | No extra dependencies, scoped styles, simple |
| Routing | **React Router v7** (hash router) | Hash routing works on GitHub Pages without 404.html hacks |
| Package manager | **npm** | Standard, no extra tooling |

No backend, no API calls, no authentication. The SPA reads static JSON files from `data/` at runtime via `fetch()`.

## 6. Pages and Components

### Overview (`#/`)

- **Header**: "TuringTrader â€” Algorithm Evaluation"
- **Metric cards row**: Best Sharpe, best return, lowest drawdown (across risk levels) â€” highlighting which risk level achieves each
- **Comparison table**: All risk levels side by side with key metrics (return, Sharpe, drawdown, win rate, trades). Sortable by column. Generated from `summary.json`
- **Quick equity curve**: Small multi-line Plotly chart showing all risk levels overlaid (normalized to $100k start)

### Equity Curves (`#/equity`)

- **Risk level selector**: Checkboxes or multi-select to toggle which risk levels are displayed
- **Main Plotly chart**: Interactive line chart of daily portfolio value over time
  - Zoom, pan, hover with date + balance
  - Toggle between absolute ($) and relative (%) views
- **Drawdown chart below**: Underwater chart showing drawdown % over time for selected risk levels

### Risk Analysis (`#/risk`)

- **Risk parameters table**: Shows how `RiskParameters` vary across levels (max daily risk, position size, stop loss, etc.) â€” sourced from `risk_level_{N}.json` `risk_parameters` field
- **Bar charts**: Side-by-side comparison of Sharpe, Sortino, Calmar, max drawdown, win rate across risk levels
- **Return distribution**: Histogram of daily returns for a selected risk level

## 7. Data Generation Script

**New file: `scripts/generate_dashboard_data.py`**

This script runs the existing backtesting framework and writes JSON to `dashboard/public/data/`.

```python
"""
Generate static JSON data for the GitHub Pages dashboard.

Usage:
    python scripts/generate_dashboard_data.py [--test-mode] [--period DAYS]

Reads from the existing backtesting framework. Writes JSON to dashboard/public/data/.
"""
```

Implementation approach:
1. Import `BacktestEngine` from `backtesting/backtest_engine.py`
2. Import `PerformanceAnalyzer` from `backtesting/performance_analyzer.py`
3. Run backtests for risk levels 1â€“10 (using `--test-mode` for mock data or real yfinance data)
4. Call `PerformanceAnalyzer.analyze_results()` and `compare_risk_levels()` for enriched metrics
5. Serialize results to JSON files matching the schema in Section 5
6. Include `risk_parameters` from `ibkr_trader.config.RiskParameters` for each level

This script is run manually (or in a separate workflow) and results are committed. The GitHub Pages deploy workflow does NOT run this script â€” it only builds the React app.

## 9. GitHub Actions Workflow

**New file: `.github/workflows/deploy-dashboard.yml`**

```yaml
name: Deploy Dashboard to GitHub Pages

on:
  push:
    branches: [main]
    paths:
      - 'dashboard/**'
  workflow_dispatch:

permissions:
  contents: read
  pages: write
  id-token: write

concurrency:
  group: pages
  cancel-in-progress: true

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: 22
          cache: npm
          cache-dependency-path: dashboard/package-lock.json

      - run: npm ci
        working-directory: dashboard

      - run: npm run build
        working-directory: dashboard

      - uses: actions/upload-pages-artifact@v3
        with:
          path: dashboard/dist

  deploy:
    needs: build
    runs-on: ubuntu-latest
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    steps:
      - id: deployment
        uses: actions/deploy-pages@v4
```

Key points:
- Only triggers on changes to `dashboard/` (not trading engine changes)
- No Python, no secrets, no database â€” pure static build
- `workflow_dispatch` allows manual re-deploy

## 11. Implementation Order

1. Create `scripts/generate_dashboard_data.py` â€” generate JSON from backtests

8. Build `EquityCurve` page (Plotly line chart + drawdown chart)
9. Build `RiskAnalysis` page (parameter table + bar charts + histogram)

### Phase 3: Deploy
10. Add Vite config with `base: '/turingtrader/'`
11. Add GitHub Actions workflow
12. Enable GitHub Pages in repo settings
13. Verify at `https://mazharm.github.io/turingtrader/`

## 12. What This Spec Does NOT Cover

- **Live trading dashboard**: The Docker-based Dash app remains the live monitoring tool
- **Real-time data**: The SPA shows pre-computed results only
- **Authentication**: The site is public
- **Parameter sweep UI** (phase 2): Could add a page showing `parameter_sweep.json` results later
- **Automated data refresh**: Running `generate_dashboard_data.py` on a schedule could be added as a separate workflow later if desired
